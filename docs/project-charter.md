# プロジェクト憲章

- ドキュメント種別: プロジェクト憲章（Project Charter）
- プロジェクト名: タスク管理アプリ（リポジトリ名: `spec-driven-dev`）
- 最終更新: 2026-07-05
- ステータス: ドラフト v0.1

---

## 1. 目的（Purpose）

個人が日々のタスクを **素早く登録し、状態を管理し、整理できる** 軽量なWebアプリケーションを提供する。
あわせて、本リポジトリを **仕様駆動開発（Spec-Driven Development）の実践例** として位置づけ、
ドキュメントとコードが常に整合した状態で継続的に改変できる開発基盤を確立する。

## 2. 背景（Background）

- 初期実装はブラウザの `localStorage` に保存する単一ページアプリだった。
- 複数端末での利用やデータの永続的な保全のため、**サーバー永続化（RDB）** が必要になった。
- そこで Python バックエンド + SQLite を追加し、フロントエンドをAPI連携型へ移行した（v0.1）。
- 今後の機能拡張に備え、判断の拠り所となるステアリングドキュメント群を整備する。

## 3. スコープ（Scope）

### 3.1 対象に含む（In Scope）

- タスクの作成・一覧・完了状態の切り替え・削除。
- 完了済みタスクの一括削除。
- 一覧のフィルタ表示（すべて / 未完了 / 完了）と残件数の表示。
- REST API バックエンドと RDB による永続化。

### 3.2 対象に含まない（Out of Scope, v0.1時点）

- ユーザー認証・アカウント管理・マルチテナント。
- タスクの期限・タグ・優先度・並び替え・検索。
- リアルタイム同期、通知、モバイルネイティブアプリ。
- 本番デプロイ基盤（CI/CDパイプライン、監視、スケーリング）。

> これらは将来の拡張候補。着手時に PRD / 要求仕様を更新してからスコープへ昇格させる。

## 4. 成功基準（Success Criteria）

| 区分 | 基準 |
|---|---|
| 機能 | 追加・切替・削除・一括削除・フィルタが期待通り動作する |
| 永続性 | ブラウザ再読込・サーバー再起動後もデータが保持される |
| 独立性 | 外部依存パッケージなしでバックエンドが起動できる |
| 整合性 | 実装と docs 配下のドキュメントが一致している |
| 開発体験 | `npm run server` と `npm run dev` の2コマンドで起動できる |

## 5. 主要成果物（Deliverables）

- 稼働するフロントエンド（React/Vite）とバックエンド（Python/SQLite）。
- `docs/` 配下のステアリングドキュメント一式。
- 起動・開発手順（README および開発ガイドライン）。

## 6. 制約・前提（Constraints & Assumptions）

- **バックエンドは Python 標準ライブラリのみで実装する**（外部パッケージを増やさない方針）。
- RDB は導入容易性を優先し SQLite を採用する（将来 PostgreSQL 等へ差し替え可能な設計とする）。
- 実行環境のシステム Python は標準ライブラリが欠落しているため、
  自己完結型 Python を `.python/` に配置して利用する（`.gitignore` 済み、配布物には含めない）。
- 単一ユーザー・単一プロセス運用を前提とする（同時多人数アクセスは想定外）。

## 7. 体制・役割（Roles）

小規模開発を想定し、一人が複数の役割を兼務する。

| 役割 | 責務 |
|---|---|
| プロダクトオーナー | スコープ・優先度の決定、PRD の承認 |
| 開発者 | 設計・実装・テスト・ドキュメント更新 |
| レビュアー | 変更内容と仕様の整合性チェック |

## 8. リスクと対応（Risks）

| リスク | 影響 | 対応方針 |
|---|---|---|
| 実行環境の Python 標準ライブラリ欠落 | バックエンドが起動不能 | 同梱ランタイム `.python/` を利用、`run.sh` で自動切替 |
| SQLite の同時書き込み制約 | 並行アクセスで競合 | 単一ユーザー前提を維持。必要時に RDBMS を切替 |
| ドキュメントとコードの乖離 | 判断ミス・手戻り | 変更時に docs を同期更新するルールを徹底 |
