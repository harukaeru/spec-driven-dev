# 開発ガイドライン

- ドキュメント種別: 開発ガイドライン（コーディング規約・運用ルール）
- 最終更新: 2026-07-05
- ステータス: ドラフト v0.1

---

## 1. 基本原則

1. **仕様が先、実装が後**: 仕様に影響する変更は、先に `docs/` を更新して合意する。
2. **ドキュメントとコードを一致させる**: 乖離を見つけたら小さくてもその場で直す。
3. **依存を増やさない**: バックエンドは Python 標準ライブラリのみ。フロントの新規依存は必要性を明記して追加。
4. **周囲のコードに合わせる**: 命名・コメント量・スタイルは既存コードに倣う。

## 2. 開発環境のセットアップ

```bash
# 依存インストール（フロントエンド）
npm install

# バックエンド起動（http://localhost:8000）
npm run server

# フロントエンド起動（http://localhost:5173）※別ターミナル
npm run dev
```

- バックエンドは `backend/run.sh` 経由で起動する。`.python/`（同梱ランタイム）があれば優先し、
  無ければシステムの `python3` を使う。
- 開発時、ブラウザからの `/api/*` リクエストは Vite が `:8000` へプロキシする（同一オリジン扱い）。

## 3. コーディング規約

### 3.1 共通
- インデントは既存ファイルに合わせる（JS: スペース2、Python: スペース4）。
- 文字コードは UTF-8。UI・メッセージは日本語を基本とする。
- 命名は意味の分かる名前を使い、短縮しすぎない。

### 3.2 フロントエンド（React / JavaScript）
- 関数コンポーネント + Hooks を用いる。クラスコンポーネントは使わない。
- 状態は必要最小限に保ち、当面は `App.jsx` へ集約する（過度な分割をしない）。
- API 呼び出しは `fetch` を用い、エラーは try/catch で捕捉して `error` ステートに反映する。
- 副作用（初期ロード等）は `useEffect`、派生値は `useMemo` を用いる。
- 直接 `localStorage` にドメインデータを保存しない（永続化はサーバー経由）。

### 3.3 バックエンド（Python）
- 標準ライブラリのみを使用する（`http.server`, `sqlite3`, `json` 等）。
- **SQL は必ずパラメータ化クエリ**（`?` プレースホルダ）で書く。文字列連結でのSQL組み立て禁止。
- HTTP 処理・ルーティング・DB操作は関数単位で分離し、ハンドラを薄く保つ。
- レスポンスは JSON（`ensure_ascii=False` で日本語をそのまま返す）。
- ステータスコードは要求仕様（[requirements.md](./requirements.md) §2）に従う。

## 4. API 変更のルール

- エンドポイントの追加・変更時は、先に[要求仕様 §2](./requirements.md) と
  [アーキテクチャ設計書](./architecture.md)の API 節を更新する。
- 後方互換を壊す変更は理由と移行方針を記載する。
- レスポンス形状（Task リソース）を変える場合はフロントの `App.jsx` も同時に更新する。

## 5. データベース変更のルール

- スキーマは `backend/server.py` の `init_db()` に定義する。
- 破壊的なスキーマ変更を行う場合は、マイグレーション方針（既存 `tasks.db` の扱い）を明記する。
- `backend/tasks.db` はデータ実体であり **コミットしない**（`.gitignore` 済み）。

## 6. Git 運用

- `main` へ直接コミットせず、作業用ブランチを切る（例: `feat/...`, `fix/...`, `docs/...`）。
- コミットは論理単位で小さく。メッセージは変更意図が分かる日本語または英語で記述する。
- コミット/プッシュはユーザーの指示があるときに行う。
- 生成物・ローカル成果物（`node_modules/`, `dist/`, `.python/`, `*.db`）はコミットしない。

## 7. レビュー観点

- 要求仕様の受け入れ基準を満たしているか。
- ドキュメント（docs/）が実装と一致しているか。
- SQL がパラメータ化されているか（インジェクション対策）。
- 不要な依存を追加していないか。
- エラーハンドリングとステータスコードが仕様通りか。

## 8. テスト・検証

- 自動テストは未整備（v0.1）。当面は手動検証を必須とする。
- 変更後は最低限、以下を確認する。
  - フロント起動 → 追加・切替・削除・一括削除・フィルタが動作する。
  - 再読込・サーバー再起動後もデータが保持される。
- API 単体は `curl` で確認できる（例）。

```bash
curl -s -X POST localhost:8000/api/tasks -H 'Content-Type: application/json' -d '{"title":"test"}'
curl -s localhost:8000/api/tasks
```

- 将来: バックエンドは `unittest`（標準ライブラリ）でのテスト追加を推奨。

## 9. ドキュメント更新のトリガ

以下の変更時は対応ドキュメントを必ず更新する。

| 変更内容 | 更新対象 |
|---|---|
| 機能の追加/変更 | PRD / 要求仕様 |
| API/データモデル変更 | 要求仕様 / アーキテクチャ設計書 |
| ディレクトリ構成変更 | リポジトリ構造定義書 |
| 技術選定・方針変更 | アーキテクチャ設計書 / プロジェクト憲章 |
