# ステアリングドキュメント

本プロジェクト「タスク管理アプリ（spec-driven-dev）」の開発・改変を進めるうえで、
常に参照すべき方針・前提・成果物をまとめた **中心となる案内図（インデックス）** です。
新しい機能追加や改修に着手する前に、まず本書を起点に関連ドキュメントを確認してください。

- 最終更新: 2026-07-05
- ステータス: ドラフト（v0.1 時点の実装に基づく）

---

## 1. このドキュメント群の目的

Spec-Driven Development（仕様駆動開発）を実践するため、
「何を作るか（Why / What）」と「どう作るか（How）」を文書として固定し、
実装より先に合意を形成することを目的とします。

コード変更のたびに判断がぶれないよう、以下を単一の情報源（Single Source of Truth）とします。

## 3. プロダクト概要（要約）

- **プロダクト**: ブラウザで動作するシンプルなタスク管理アプリ。
- **提供価値**: タスクの追加・完了切り替え・削除・フィルタ表示を、サーバー永続化つきで提供する。
- **現状（v0.1）**: React（Vite）製のフロントエンドと、Python 標準ライブラリのみで実装した
  REST API バックエンド、RDB（SQLite）による永続化まで実装済み。

## 4. 技術スタック（要約）

| レイヤ | 採用技術 | 補足 |
|---|---|---|
| フロントエンド | React 18 + Vite 5 | SPA。状態は `App.jsx` に集約 |
| バックエンド | Python 3.12（標準ライブラリのみ） | `http.server` + `sqlite3`。外部依存なし |
| データベース | SQLite | RDB。`backend/tasks.db` に永続化 |
| 通信 | REST / JSON | 開発時は Vite が `/api` をバックエンドへプロキシ |

詳細と選定理由は[アーキテクチャ設計書](./architecture.md)を参照。

## 5. 改変作業の進め方（Spec-Driven のワークフロー）

1. **要求の確認**: 変更対象が[要求仕様](./requirements.md)・[PRD](./prd.md)のどこに該当するか特定する。
2. **仕様の更新**: 仕様に変更が及ぶ場合、実装より先に該当ドキュメントを更新し合意する。
3. **実装**: [開発ガイドライン](./development-guidelines.md)と[リポジトリ構造定義書](./repository-structure.md)に従う。
4. **検証**: 受け入れ基準（要求仕様の各項目）を満たすことを確認する。
5. **ドキュメント同期**: 実装に伴い変わった構成・APIを各ドキュメントへ反映する。

> 原則: **「ドキュメントとコードは常に一致させる」**。乖離を見つけたら小さくてもその場で直す。

## 6. 用語

| 用語 | 意味 |
|---|---|
| タスク | ユーザーが管理する単一の作業項目。`title` と `completed` を持つ |
| フィルタ | 一覧の表示条件（すべて / 未完了 / 完了） |
| 同梱ランタイム | リポジトリ直下 `.python/` に配置した自己完結型 Python（環境依存回避用） |
