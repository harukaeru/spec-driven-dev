# 要求仕様

- ドキュメント種別: 要求仕様書（機能要求 / 非機能要求）
- 最終更新: 2026-07-05
- ステータス: ドラフト v0.1
- 上位文書: [PRD](./prd.md) / [プロジェクト憲章](./project-charter.md)

各要求は検証可能な形で記述する。表記は次の通り。

- **MUST**: 必須。満たさなければ不合格。
- **SHOULD**: 推奨。合理的理由なく外さない。
- **MAY**: 任意。

---

## 1. 機能要求（Functional Requirements）

### FR-01 タスク追加（PRD: F-01）
- MUST: 入力欄にテキストを入力し「追加」または Enter で送信するとタスクが作成される。
- MUST: タイトルは前後の空白を除去（trim）し、空文字のみの場合は作成しない。
- MUST: 作成されたタスクは一覧の **先頭（新しい順）** に表示される。
- MUST: 作成後、入力欄はクリアされる。
- **受け入れ基準**: 「牛乳を買う」を入力→追加すると、先頭に未完了状態で表示され、入力欄が空になる。

### FR-02 タスク一覧表示（PRD: F-02）
- MUST: 起動時にサーバーから全タスクを取得し表示する。
- MUST: 並び順は作成が新しいものが上（`id` 降順）。
- SHOULD: 取得中は「読み込み中」を表示する。
- MUST: タスクが0件のときは「タスクはありません」を表示する。

### FR-03 完了状態の切替（PRD: F-03）
- MUST: 各タスクのチェックボックス操作で `completed` を反転する。
- MUST: 完了タスクは視覚的に区別（取り消し線等）される。
- MUST: 変更はサーバーへ永続化される。
- **受け入れ基準**: チェック→再読込しても完了状態が保持される。

### FR-04 タスク削除（PRD: F-04）
- MUST: 各タスクの「×」で当該タスクを削除する。
- MUST: 削除はサーバーへ反映され、一覧から消える。

### FR-05 完了済み一括削除（PRD: F-05）
- MUST: 「完了済みを削除」で `completed = true` のタスクを全削除する。
- MUST: 未完了タスクは残る。
- SHOULD: 削除件数を把握できる（APIは削除数を返す）。

### FR-06 フィルタ表示（PRD: F-06）
- MUST: 「すべて / 未完了 / 完了」の3種で一覧を絞り込める。
- MUST: 選択中のフィルタが視覚的に分かる。
- MAY: フィルタ状態の永続化は不要（再読込で「すべて」に戻ってよい）。

### FR-07 残件数表示（PRD: F-07）
- MUST: 未完了タスク数を「残り N 件」として常時表示する。

### FR-08 永続化（PRD: F-08）
- MUST: 全データは RDB（SQLite）に保存される。
- MUST: サーバー再起動後もデータが保持される。

### FR-09 通信状態・エラー表示（PRD: F-09）
- SHOULD: API 呼び出し失敗時にエラーメッセージを画面へ表示する。
- SHOULD: エラー表示は次の成功操作で解消される。

## 2. API 要求（Interface Requirements）

ベースパス: `/api`。リクエスト/レスポンスは JSON（UTF-8）。

| メソッド | パス | 用途 | 成功時 |
|---|---|---|---|
| GET | `/api/tasks` | 一覧取得 | 200 + タスク配列 |
| POST | `/api/tasks` | 作成 `{title}` | 201 + 作成タスク |
| PATCH | `/api/tasks/{id}` | 更新 `{title?, completed?}` | 200 + 更新タスク |
| DELETE | `/api/tasks/{id}` | 削除 | 204 |
| DELETE | `/api/tasks/completed` | 完了済み一括削除 | 200 + `{deleted: N}` |

### タスクリソース（Task）

```json
{ "id": 1, "title": "牛乳を買う", "completed": false, "created_at": "2026-07-05 10:00:00" }
```

### エラー要求
- MUST: `title` が空 → **422**（`{"error": ...}`）。
- MUST: 不正なJSONボディ → **400**。
- MUST: 存在しない `id` への PATCH/DELETE → **404**。
- MUST: 未定義ルート → **404**。

## 3. 非機能要求（Non-Functional Requirements）

### NFR-01 性能
- SHOULD: 個人利用規模（〜数百件）で、各操作の応答が体感遅延なく完了する。

### NFR-02 可搬性・依存
- MUST: バックエンドは **Python 標準ライブラリのみ** で動作し、外部パッケージ導入を要しない。
- MUST: `npm run server` と `npm run dev` の2コマンドで開発環境が起動する。

### NFR-03 データ整合性
- MUST: `title` は NOT NULL。`completed` は真偽（0/1）で保持。
- MUST: `id` は自動採番の一意キー。

### NFR-04 保守性
- SHOULD: フロントの状態管理は単一コンポーネント（`App.jsx`）に集約し追跡しやすくする。
- SHOULD: バックエンドはルーティング・DB操作・HTTP処理を関数単位で分離する。

### NFR-05 セキュリティ（現状の前提）
- 単一ユーザー・ローカル利用前提のため認証は持たない。
- MUST: SQL は必ずプレースホルダ（パラメータ化クエリ）で発行し、文字列連結を禁止する。
- 注意: 現状の CORS は全許可（`*`）。外部公開時は要見直し（[アーキテクチャ設計書](./architecture.md)参照）。

### NFR-06 可用性・運用
- 単一プロセス運用を前提とする。プロセス停止＝サービス停止。
- SQLite ファイル（`backend/tasks.db`）がデータの実体であり、バックアップ対象とする。
